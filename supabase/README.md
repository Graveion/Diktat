# Diktat — Supabase (accounts, machines, pairing)

Identity and the machine registry live in Supabase. This directory is the config
+ migrations, versioned in-repo; we apply changes with the Supabase CLI.

## What lives here

- `config.toml` — project config (generated by `supabase init`).
- `migrations/` — ordered SQL. `0001_accounts_machines_pairing.sql` is the schema.

## Model

- **Account = `auth.users`** (Supabase Auth via Sign in with Apple + Google).
  `accountId == auth.users.id`. No separate accounts table.
- **`public.machines`** — one row per dev machine. `token_hash` = sha256 of the
  per-machine daemon token (plaintext shown once, at pairing, never stored).
- **`public.pairing_codes`** — short-lived codes the phone mints and the daemon
  redeems.

This replaces the static `relay/relay-config.json`:
| old | new |
|---|---|
| `machines{id -> {daemonToken, accountId}}` | `public.machines` (token hashed) |
| `accountTokens{token -> accountId}` | the Supabase **JWT** (`sub` = accountId) |

## Auth at the relay

- **Client leg (phone):** bearer is the Supabase **access-token JWT**. The relay
  verifies it (JWT secret / JWKS), takes `sub` as the accountId. No DB hit.
- **Agent leg (daemon):** bearer is the per-machine daemon token. The relay
  hashes it and matches `machines.token_hash` (service-role read).
- **Ownership gate:** `machines[machineId].account_id === jwt.sub`.

## Pairing flow

1. Phone (signed in) inserts a `pairing_codes` row for itself (RLS: own only),
   shows the code.
2. User runs `diktat pair <code>` on the Mac. Daemon → relay `POST /pair/complete`
   with `{ code, machineId, name }`.
3. Relay (service role): validate code (unexpired, unconsumed) → generate daemon
   token → insert `machines` row with `token_hash` + the code's `account_id` →
   mark code consumed → return the **plaintext token once** to the daemon, which
   writes it to its config.
4. Daemon connects `/agent` with that token.

## Applying changes

```bash
supabase link --project-ref <ref>      # one-time, needs access token
supabase db push                       # apply migrations to the linked project
```

## Keys (where each is used)

| Key | Used by | Secret? |
|---|---|---|
| Project ref | `supabase link` | no |
| `anon` key + project URL | the app (Supabase client) | public |
| `service_role` key | the relay (bypass RLS for pairing/auth) | **secret** |
| JWT secret / JWKS URL | the relay (verify phone JWTs) | **secret** |

Secrets go in the relay's env / a gitignored `.env`, never in the repo.
